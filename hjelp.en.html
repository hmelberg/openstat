OpenStat

Write and run Python, R and SQL directly in the browser — no installation required.

What is this?

OpenStat is an engine for asking questions that can be answered with code and statistics — and for seeing and revising the code behind the answer. Write Python, R or SQL, see results immediately, and run hybrid scripts where different languages work together.

Note: This is a hobby project and is not made by microdata.no itself. The page does not contain real figures, and there is no guarantee that the analyses are correctly implemented.

Everything happens in the browser — no data leaves your machine.

Quick start: Write code in the text field, press Ctrl+Enter or the Run button. Use Tab for suggested commands and variables.

Running

Running directly

Press Ctrl+Enter or the Run button. Results are shown in the output field. Tables, charts and text are displayed automatically.

Stop / cancel

Press the Run button again while running to cancel.

Multi-key: only ONE variable in by() and on. Create a composite key first: generate composite = string(k1) ++ "_" ++ string(k2).

Pseudonym variables

Variables that identify units (typically variables with names ending in _FNR — e.g. BEFOLKNING_MOR_FNR) are pseudonyms. In microdata.no these can only be used as a key in collapse(by) or merge(on) — not in generate/replace, comparisons or other transformations.

Control flow

For loops use the prod grammar: a range with n : m (inclusive), or a value list (comma- or whitespace-separated). Use $i for substitution. Do not use parentheses around the iterator list or a literal ellipsis ....

for i in 2018 : 2020
  import db/INNTEKT_WLONN $i-01-01 as inntekt_$i
end

for forelder in mor, far
  // ...
end

let maks = 200000
generate hoy = (inntekt > maks)

Disclosure control

m2py emulates microdata.no's rules by default — both syntax/validation (so your script runs in prod) and suppression of output (so you see the same numbers as in prod). You can turn everything off in the hamburger menu ("Disclosure control: OFF") to experiment without restrictions, but then you risk writing scripts that won't run in microdata.no. Specific rules:

  • Pseudonym validation. Variables ending in _FNR can only be used as a key in collapse(by) or merge(on) — not in generate, replace, comparisons, etc.
  • Type check. Alphanumeric variables (e.g. BEFOLKNING_KJOENN) are rejected for numeric operations such as min, max, mean, sum, median.
  • For loops: use $i substitution and either a range (1998 : 2009) or a comma/whitespace list. Parentheses around values and a literal ... are not valid.
  • Missing values. Assignment with . is OK (generate x = .). Comparison with ., however, is not valid — keep if y == . is rejected in microdata.no. Use sysmiss(y) to test whether a value is missing.
  • Minimum population 1000. keep if, drop if and sample cannot leave fewer than 1000 units.
  • Tables with too many small cells are suppressed. tabulate is blocked if more than 50% of the cells have a frequency lower than 5.
  • Changes must affect ≥10 units. generate, replace and recode are rejected if the change affects 1–9 rows (or all except 1–9).
  • Descriptive statistics require ≥10 units. summarize, correlate, ci, anova, normaltest are blocked for populations under 10.
  • Median/percentiles are shown with 3 significant digits. In summarize, for example, 47,238 becomes 47,200 and 2.7183 becomes 2.72.
  • Winsorization 1/99%. Numeric variables are capped at the 1st and 99th percentile before displaying the mean and standard deviation in summarize, and before plotting in histogram, scatter and boxplot. The median and quartiles are not affected. Regression and collapse use raw data.

Per-script override

You can also control disclosure control from within the script itself with a magic directive at the top of the code. The directive is an ordinary comment (//) that is ignored by microdata.no production, but is read by m2py before execution and switched back afterwards (does not affect other scripts):

// m2py: disclosure-control=off
create-dataset bef
import db/BEFOLKNING_KJOENN as kjonn
summarize kjonn

Values: on/off, true/false, 1/0, yes/no. The short form dc is also accepted: // m2py: dc=off. Useful in API/script usage where you want to control the mode without changing global settings.

Data minimisation assessment

m2py can ask an AI to assess whether your script practices data minimisation — the principle of retrieving and using only the minimum amount of data needed for the research question. The feature is advisory; the final assessment rests with the researcher and data controller.

The assessment is grounded in:

  • GDPR Art. 5(1)(c) — data minimisation
  • Norwegian Health Registry Act (Helseregisterloven) § 6 — degree of personal identification
  • GDPR Art. 89(1) — scientific research
  • GDPR Art. 9 — special categories (sensitive data)

How it works

Click the hamburger menu → "Assess data minimisation". The first time you use the feature, a confirmation is shown that the script is sent to Anthropic. Actual microdata values are never sent.

In the options modal you can:

  • Choose language (auto / microdata / R / Python) — auto detects from the script
  • Choose report length: Short (quick observations) or Long (all dimensions with follow-up questions)
  • Request a revised script — the AI appends a conservative revised version at the end of the answer, which you can copy or replace the editor with
  • Specify purpose and context — optional; gives a more precise assessment

Login

The AI assessment requires that you are logged in to m2py. Use the Log in button at the top and follow the link in the email. Once you are logged in (via an FHI or other approved email address), you have access to the data minimisation feature automatically — no separate access key is needed.

Sensitive variables

The AI automatically flags variables that fall under special categories (GDPR Art. 9): ethnicity, religion, sexual orientation, particularly sensitive health information (abortion, sexually transmitted diseases, psychiatry, etc.), and legal/criminal records. These require a higher justification threshold.

Privacy comments in the script

You can add structured justifications directly in the script: 

// personvern blokk start
// purpose: Study the relationship between education and income
// time period: 1970-1980 because the cohort should have completed education
// geography: municipality needed to see regional differences
// sensitive groups: no
// alternatives considered: SSB table A-04 was too coarse-grained
// personvern blokk slutt

Or free text: 

// personvern blokk start
// Purpose: study the relationship between education and income for the 1970-1980 cohort.
// The time period was chosen because the cohort should have completed education by the end of the analysis.
// Municipality needed for regional differences.
// personvern blokk slutt

If you check "Save as privacy block" in the secondary modal, m2py writes the content into the script automatically after the assessment.

Single lines also work: 

// personvern: dates truncated to month to avoid unnecessary precision

You can also use # instead of // in Python/R scripts.

Directives in the script

You can control the AI behavior directly in the script: 

// personvern: revider-script: ja

This makes the AI turn on revised-script output regardless of what you choose in the UI. Useful if you have a template that should always be revised.

Save, share and fetch scripts

All the options below are in the hamburger menu at the top left. They run entirely in your browser — we never store your scripts on a server.

Download code / Load file

Download code downloads the script as a .txt file to your machine. Load file (local) opens such a file again. This is purely local file handling.

Share (copy link)

Creates a link that contains the entire script, compressed into the address itself (after #). The link is copied to the clipboard — send it by email, Slack, or paste it wherever you like. Whoever opens the link gets the script directly into the editor.

  • Nothing is stored with us, and the recipient does not need to log in.
  • The link is a frozen snapshot — if you change the script afterwards, you must share a new link.
  • Suitable for scripts up to a few thousand characters. If the script is too large for a URL, you'll be notified — use "Download code" (file) or GitHub instead.

Open from URL

Paste a direct link to a text file, and the content is fetched into the editor. The most recent URLs you've used are remembered locally for quick reopening.

  • Works with GitHub "raw" links, GitHub gist (raw) and Dropbox.
  • Regular share links (e.g. a Google Drive or GitHub page) do not work — you must use the raw file link. If you get an error message, it's almost always because the source doesn't allow direct fetching (CORS).
  • The file extension controls the mode: .py → Python, .r → R, otherwise Microdata.

GitHub (advanced)

Lets you save and fetch scripts directly in your own GitHub repo — with version history and the ability to collaborate. Everything is under the GitHub submenu in the hamburger menu.

1. Settings (one-time). Fill in a personal access token (PAT), repo (owner/name) and branch. Press Test connection to confirm, and Save settings. The information is remembered locally in the browser. You do not need to specify any file path here — you're connecting to the repo, not a single file.

Multiple profiles. If you have several repos or accounts (e.g. work and personal), you can save them as separate profiles and switch between them in the dropdown at the top of the dialog ("New" to add, "Delete" to remove). The selected profile controls Open/Save. With Show and Copy you can view or copy the token — useful when setting up the same profile on another machine.

How to create a token: GitHub → Settings → Developer settings → Fine-grained tokens → Generate new token. Give it access to only the relevant repo, and set Repository permissions → Contents → Read and write. Copy the token (shown only once) and paste it into the app. Feel free to set an expiry date; you can revoke the token at any time on the same page.

2. Daily use. Once the connection is set up:

  • Open file… shows a file picker with all the files in the repo. Filter in the search field and click a file to fetch it into the editor.
  • Save writes to the file you last opened or saved (the current file, marked with a cloud symbol ☁ next to the script name). If you don't have a current file yet, it behaves like "Save as". If you open a new or different script (New, local file, example, link or URL), the cloud symbol is disconnected so you don't accidentally overwrite the wrong file.
  • Save as… lets you type a new file path (existing folders are suggested). The .txt extension is added automatically if you don't type one.
  • Refetch fetches the current file again from GitHub (reads — does not save). Unsaved local changes are lost, so you'll be asked to confirm.
  • One token, many files: the token applies to the whole repo — you don't need a token per file.
  • Public vs. private is controlled via the repo's own setting on GitHub — not in the app.
  • Sharing: if the script is in a public repo, you just share the raw file link; the recipient uses "Open from URL".
  • The token is a secret and is stored in plain text in your browser. Use a shared machine with care, and revoke the token if you lose access to the machine.

Recently used

A small down-arrow icon (▾) next to the filename shows the most recent GitHub and URL scripts you've opened, so you can fetch them again with one click without typing the whole address. (Local files cannot be reopened by the browser.)

Where is everything stored — and does it work on multiple machines?

It's useful to distinguish the file from the machine:

  • Scripts stored in GitHub live in the cloud and are accessible from anywhere. On a new machine you just paste in the same token once (or nothing if the repo is public — then you use "Open from URL").
  • Share links carry the script within themselves and work on any machine.
  • What is tied to one machine is only local things in the browser: the list of "recent URLs" and the pasted-in GitHub token. Your actual scripts in GitHub are not locked to the machine.

Run step by step — a step-by-step walkthrough

Run step by step runs the script one block at a time and reads the comments aloud using speech. All comment lines starting with // (microdata) or # (Python/R) are read aloud — the code lines are executed without commentary. Useful for presentations and teaching.

Blocks

A block is a group of consecutive lines with no blank lines between them. One or more blank lines separate blocks.

// This comment is read ALOUD before the code runs
create-dataset df
import db/BEFOLKNING_KJOENN as kjonn

// This is the next block
summarize kjonn
// This comment is read after the result is shown

Comment syntax

SyntaxModeEffect
// textMicrodataRead as speech; a leading comment is read BEFORE the code, a trailing one is read AFTER
# textPython RSame as above
//- or #-AllSilent comment — the first code word is not read aloud, and text after - is not read either
//(lang=nb-NO,rate=1.2) textAllSpeech with custom parameters (language, rate, voice, pitch)

Pause, navigation and breakpoints

  • Press Run step by step again to pause/resume
  • The arrow buttons jump to the previous/next block
  • Breakpoints (click a line number) make the run pause automatically

Widgets in Run step by step

Widgets are interactive cards shown while the script runs step by step. They are inserted as special comment lines and are available in all modes.

Syntax

// microdata:
//widget NAME { ...json... }

# python or R:
#widget NAME { ...json... }

Block syntax for longer content

For HTML, SVG and markdown with multiple lines, you can use block syntax instead of JSON:

//start_widget:html(title="Example", speak="Here you see HTML content.")
<div style="text-align:center">
  <h2>Hello from HTML</h2>
  <p>Multiple lines without escaping.</p>
</div>
//end_widget

Available widgets

WidgetExampleDescription
title //widget title {"text":"Chapter 1","size":"full"} Large title card with animation. size: "full", "medium", "small"
markdown //widget markdown {"text":"# Hello\nThis is **bold** text"} Rendered Markdown content
question //widget question {"q":"What is the p-value?","options":["<0.05",">0.05"],"answer":0} Multiple-choice question with feedback. "require_correct": true gives a try-again mode

All widgets support speak="text" for parallel speech narration while the widget is shown, and auto_advance_ms=3000 for automatic closing after a specified time.

AI assistant

An experimental side panel with an AI assistant that can answer questions about microdata syntax, suggest variables, and generate or revise scripts. Opened with the star icon at the top right or Ctrl+I.

Requires your own Anthropic API key. Click the gear icon in the AI panel and paste in a key from console.anthropic.com. The key is stored only locally in the browser (localStorage) and sent as an X-Anthropic-Key header straight through to Anthropic for each request.

What the assistant can do

The assistant classifies the question and answers in one of three ways:

  • Question — explains commands, syntax and usage ("What does reshape long do?")
  • Variable search — finds relevant variables in the microdata catalogue ("What variables exist for education?")
  • Script generation — creates a complete microdata script from a natural-language question ("Show summary statistics for income and gender")

Revising an existing script

To let the assistant modify the script in the editor, you must check "Include script from editor (revise)" right above the text field before sending. This uses the content of the editor as a starting point, and the request is interpreted as a revision — for example "Add log income" or "Filter on age over 25".

Without this checkbox checked, a new script is created from scratch, even if the editor has content.

Buttons on generated scripts

When the assistant returns a script, three buttons are shown under the code block:

  • 📋 Copy — copies the script to the clipboard
  • ⬇ Replace script — replaces everything in the editor with the generated script
  • + Add — appends the script to the bottom of the editor

Validation warnings

Generated scripts are run through static validation before being shown. If there are unknown variables, unknown commands or parsing errors, these are shown as clickable chips under the script. Click a chip to compose a follow-up message asking the assistant to fix that specific error.

Time usage

Questions and variable searches usually respond in under 5 seconds. Script generation and revision can take 20–60 seconds, up to ~3 minutes for complicated requests — you'll see a "Thinking… Ns" counter while the job runs in the background.

Other controls

  • Gear icon — change API key and base URL
  • Trash icon — clear the conversation (does not delete anything on the server)
  • × — close the panel
  • Enter sends, Shift+Enter inserts a new line

Privacy

What is sent to the API is only your question text, and — if checked — the content of the script editor. Data files, run results, output and widget content are not sent. Norwegian or English language is guessed automatically from the question.

Experimental. The AI assistant can be wrong, suggest variables that don't exist, or generate scripts that don't run. Always check the result before use.

Modes

Choose a mode from the dropdown menu in the bottom bar, or use mode-switch lines in the script to switch along the way (see Hybrid scripts).

Complete R in the browser. Tidyverse, ggplot2, dplyr, survival, etc. are installed automatically on use.

Hybrid scripts

You can mix microdata, Python and R in the same script by using mode-switch lines.

Switch lines

LineEffect
## microdata / #micro / // microdataSwitch to microdata mode
## python / #pySwitch to Python mode
## r / #rSwitch to R mode

Sharing data between languages: # use

Microdata and Python blocks share datasets automatically (same engine). Between different engines — Python↔R, microdata→R, DuckDB↔others — fetch a dataset explicitly with a use line at the top of the block:

#py
df["ny"] = df["x"] * 2

#r
# use df
summary(df$ny)

Without from, the dataset is fetched from the nearest earlier block in a different language; specify # use df from python|r|duckdb when needed (required when the source is a previous run). Copy semantics: the block gets a snapshot — changes do not propagate back to the source engine.

Example: microdata + Python

## microdata
require no.ssb.fdb:52 as db
create-dataset df
import db/BEFOLKNING_FOEDSELS_AAR_MND as faarmnd
import db/INNTEKT_WLONN 2022-01-01 as inntekt
generate alder = 2022 - int(faarmnd / 100)
keep if inrange(alder, 25, 67)
keep if inntekt > 0

## python
import matplotlib.pyplot as plt
df.plot.scatter(x="alder", y="inntekt", alpha=0.3, s=8)
plt.tight_layout()
show(plt.gcf())

Example: microdata + R

# microdata
require no.ssb.fdb:52 as db
create-dataset df
import db/BEFOLKNING_KJOENN as kjonn
import db/INNTEKT_WLONN 2022-01-01 as inntekt

#r
library(dplyr)
df %>% group_by(kjonn) %>%
  summarise(snitt = mean(inntekt, na.rm = TRUE))

The datasets from the microdata section are automatically transferred to Python as pandas DataFrames and to R as data frames.
