Interactive elements in OpenStat

Three ways in, one engine underneath — pick the level of control you need.

OpenStat has three levels of abstraction for interactivity, aimed at three audiences. They are not competing systems: a #@param form, a ui.slider, and a hand-built ui.html element all render through the same engine, live in the same cells, and follow the same display rules.

LevelYou writeYou getBest for
1 — #@param a comment on an assignment line a form control that rewrites the value in your code and reruns making any script tweakable without writing UI code
2 — ui.* one line of Python a live control whose current value is returned by the call notebooks and dashboards driven by a few inputs
3 — ui.html element builders real DOM nodes with events, styling, and component libraries custom layouts, cards, buttons, anything the widgets don't cover
Choosing between #@param and ui.*: use #@param when the choice should follow the document — it writes the value back into the code line, so a saved or shared document reproduces exactly what you saw, and the exported code runs unchanged outside OpenStat. Use ui.* when the choice should follow the data or needs events — options computed at runtime (ui.dropdown(sorted(set(df["species"])))), callables instead of reruns, driving other cells with rerun="plot", animation, or values you don't want written into the document.

How output display works

One rule everywhere: a value alone on a line shows itself. Every bare expression in a cell (or script) is displayed — tables as tables, figures as figures, ui.html elements mounted into the output.

x = 5        # assignment — nothing shown
x + 1        # shows 6
"hello"      # shows hello
df.head()    # shows the table

To mute a value you don't want shown:

_tmp         # names starting with _ are never displayed
df.head();   # a trailing ; mutes any expression
n = ui.slider(0, 100)   # assignments are silent as always

Two conveniences keep interactive code quiet: None is never displayed, and a bare ui.<control>(...) call renders its control without echoing the returned number under it.

Prefer last-expression-only display (Jupyter style)? Add #options.display = last at the top of the document. #options.display = all keeps the legacy echo mode.

Level 1 — #@param forms all modes

Annotate an assignment with a #@param comment and OpenStat renders a form control for it. Changing the control writes the new value back into the code line — the form is the code — and reruns the cell (automatic by default).

#@title Iris explorer {run:"manual"}
#@markdown Pick a species and a row count, then tweak the slider below.
n       = 50        #@param {type:"slider", min:10, max:150, step:10}
species = "setosa"  #@param ["setosa", "versicolor", "virginica"]
fruit   = "apple"   #@param ["apple", "banana"] {allow-input: true}
level   = 3         #@param {type:"integer"}
start   = "2026-01-01"  #@param {type:"date"}
active  = True      #@param {type:"boolean"}
expr    = "x + 1"   #@param {type:"raw"}
  • #@title / #@markdown: a #@title line renders as a heading above the form strip and, when the cell has one, its {run:"manual"} meta becomes the cell's run default (only the first #@title in a cell counts); #@markdown lines render as prose, in the order they appear, alongside the controls. Both are Colab syntax, so pasted Colab cells render as-is.
  • Types: slider, number, integer, boolean, date, raw, plain text, and dropdowns from a list literal (add {allow-input: true} for an editable dropdown).
  • Run behavior: run:"auto" is the default — each change reruns the cell (sliders debounce). Use {run:"manual"} to only write the value; a small Run ▶ chip appears in the form until you run the cell.
  • Placement: {placement:"top"} (default), "bottom", or "left".
  • In JavaScript mode the same syntax uses //@param. Because the value is swapped into the line byte-exactly, don't end param lines with a semicolon.

Level 2 — ui.* widgets python · brython · micropython

One line per control. The call registers the control (drawn in the cell's widget strip) and returns its current value — when the user changes the control, the cell reruns and the call returns the new value. State lives outside your code, so a cell stays a pure function of its inputs (the pull model).

import ui

n       = ui.slider(1, 20, value=5, label="Rows")
species = ui.dropdown(["setosa", "versicolor"], label="Species")
strict  = ui.checkbox("Strict", value=True)
dark    = ui.switch("Dark mode")
limit   = ui.number(10, min=0, max=100, label="Limit")
name    = ui.text("iris", label="Dataset")

iris[iris["species"] == species].head(n)

Targeting reruns

rerun= decides what a change runs: "self" (default), a cell id (#%% python id=plot → rerun="plot"), a list of ids, or "none". This is how one control cell drives a separate plot cell:

#%% python
import ui
n = ui.slider(1, 50, value=10, rerun="plot")

#%% python id=plot
iris.head(n)   # reruns whenever the slider changes

Reading and steering controls from code

ui.value("n")          # read a control's stored value without running anything
h = ui.widget("n")     # handle: h.value, h.set(30), h.hide(), h.show(),
                       #         h.on("input", fn), h.element(), h.input()
  • Placement: per control (placement="left") or per cell (#%% python widgets=left).
  • sync_to="varname" pushes the value into a live session variable — seeded already at registration, then updated as the user drags, without a rerun. This makes it a full value channel of its own: ui.slider(0, 100, sync_to="n") on its own line, then use n anywhere below. One caveat: don't assign the control to the SAME name you sync to (n = ui.slider(sync_to="n")) — a rerun re-executes the assignment and the variable's type ends up depending on interaction history. Use the bare-call form, or two distinct names.
  • Widgets need a notebook (#%% cells). In a plain script a ui.* call quietly returns its default and draws nothing.
  • Pasted Jupyter code using ipywidgets also works in python mode — it is a separate compatibility bridge with its own live sync.

Level 3 — ui.html elements python · brython · micropython

Build real DOM elements from Python — around 110 standard tags via ui.html.<tag>(*children, **kwargs). An element shown as a bare expression (or via .show()) mounts into the cell's output.

ui.html.div(
    ui.html.h3("Result"),
    ui.html.p("Everything ", ui.html.b("looks good"), "."),
    cls="card", style={"padding": "1rem", "border_radius": "8px"},
)

The kwargs standard

  • cls= or class_= → the class attribute.
  • style= — a CSS string, or a dict with snake_case keys (border_radius → borderRadius).
  • data_foo= / aria_label= → data-foo / aria-label.
  • on_click=my_function (any on_<event>=) — binds a Python callable. Strings are not executed as code.
  • attrs={...} — escape hatch that sets attributes verbatim.
  • Booleans set presence-attributes (disabled=True); problems warn loudly in the console instead of failing silently.

Element methods

el = ui.html.div(id="box")
el.add("more text", ui.html.br())   # append children
el.clear()                          # empty it
el.on("click", handler)             # bind an event
el.set_style(color="red")           # style after creation
el.add_class("warn"); el.remove_class("warn")
el.show()                           # mount into the running cell's output now
el.show(target="box2")              # replace into another element by DOM id

Ready-made building blocks

ui.kpi(4.2, 0.3, unit="%", label="Growth", bra="opp")   # KPI card with delta arrow
ui.markdown("**bold** and `code`")                       # rendered markdown
ui.image(fig)                                            # matplotlib figure or URL → <img>

Component libraries — #tag.import

Pull in polished web components with one preamble line; each import gets a namespace of element builders:

#tag.import = shoelace        →  ui.sl.button("Click"), ui.sl.rating(), …
#tag.import = pico            →  ui.pico.button("Click"), …
#tag.import = https://example.com/lib.js as acme
                              →  ui.acme.button() renders <acme-button>

Containers python · brython · micropython

Widgets (level 2) live in the cell's widget strip; ui.html (level 3) composes freely but is raw. Containers close the gap: controls placed explicitly inside a layout you build — a card, a toolbar, a dashboard grid — while keeping the widgets' value store, debounce, and sync_to/rerun wiring.

Three container types, all ordinary ui.html elements underneath (nestable, stylable, .show(target=) works):

  • ui.row(gap=, wrap=, justify=, align=) / ui.column(...) — flex layouts; same kwargs, only the axis differs.
  • ui.grid(template, cols=None, rows=None, gap=None) — named areas, 1:1 with CSS grid-template-areas: | separates rows, whitespace separates columns, and . is the CSS null-cell (an empty spot in the layout).
layout = ui.grid("side main | side table", cols="220px 1fr")
panel  = ui.column(gap="0.75rem")

.add() — append or replace-in-place

Every container (and any ui.html element) has .add(child_or_list, area=, span=, align=):

  • Without area=: appends — same as passing children to the element constructor, including the list form (.add([a, b])).
  • With area=: only valid on a ui.grid(...) — targets a named area. Adding into an occupied area replaces its content rather than appending (empties it first, then renders) — this is what makes an on_change handler that redraws into an area work. Values (a DataFrame, a figure, a number) render through the same payload path on_change return values use. Adding several plain values in one .add(..., area=...) call only shows the last — the payload path is clear-then-render, not append; wrap multiple values in a container child if they should coexist.
  • span=/align= set grid-column: span N / align-self on each element child added in that call (with area=, they land on the area's cell).

The return rule — into=

Every ui.* control takes an optional into= kwarg naming the container it should mount in. It changes what the call returns:

n = ui.slider(0, 100)                 # no into= → the VALUE (pull model, as before)
s = ui.slider(0, 100, into=panel)     # into=      → the HANDLE (.value, .set(), .on())

Memorable form: without into= you get the value; with into= you get the control. rerun="self" stays the default either way — a control with an on_change handler never reruns (event-driven), and one without redraws the layout with the new value (the beginner default). Three value channels are available side by side:

ChannelSpellingSuited for
Pull (return = value)n = ui.slider(0, 100) notebooks, rerun-driven (unchanged)
Session variable (seeded sync_to) ui.slider(0, 100, sync_to="n") live value without rerun, no assignment
Handle (.value/.set/.on) s = ui.slider(0, 100, into=c) containers/apps, event-driven
#%% python
import ui

layout = ui.grid("side main | side table", cols="220px 1fr")

def update(_):
    sub = iris[iris["species"] == a.value]
    layout.add(sub.head(s.value), area="main")   # replaces area content

panel = ui.column(gap="0.75rem")
s = ui.slider(0, 100, value=30, label="Threshold", into=panel, on_change=update)
a = ui.dropdown(sorted(set(iris["species"])), into=panel, on_change=update)

layout.add(panel, area="side")
layout.add(iris.head(10), area="table")
layout

Events and reruns

Three ways to react to the user, from simplest to most explicit:

MechanismWhat happens
rerun="plot" on a widget value changes → the named cell reruns and pulls the new value
on_change=my_fn / on_click=my_fn the callable runs instead of a rerun; its return value (or printed output) is rendered into the cell's output — or into target="dom-id"
ui.on(selector, event, handler, target=None)
ui.run_cell(selector, event, cell_id)		 bind any DOM event by CSS selector — e.g. make a hand-built ui.html button run a named cell
#%% python
import ui

def compute(value):
    return ui.kpi(value * 2, label="Doubled")

n = ui.slider(0, 50, on_change=compute)   # no rerun — the KPI updates in place
